# Phase 8: Make generated planning documents Korean-first for installed skills - Context

**Gathered:** 2026-03-24
**Status:** Ready for planning
**Source:** Direct maintainer request during Phase 8 kickoff

<domain>
## Phase Boundary

Fix the installed GSD runtime so that planning documents generated by scaffold/template-driven flows come out Korean-first by default, without breaking parser-sensitive markers, command tokens, file paths, or existing automation contracts.

</domain>

<decisions>
## Implementation Decisions

### Scope of generated documents
- **D-01:** Prioritize the document-generation paths that users hit after installation, especially scaffolded and template-filled planning docs.
- **D-02:** Cover both runtime generation code and the reusable templates that guide downstream doc creation.

### Compatibility guardrails
- **D-03:** Preserve machine-sensitive tokens and labels that existing parsers or workflows depend on, even when the surrounding prose becomes Korean-first.
- **D-04:** Treat regression tests as part of the deliverable so the Korean-first behavior is locked in.

### Claude's Discretion
- Exact grouping of files and whether the work is split into one or more plans.
- Which generated-doc templates beyond the core scaffold/template entrypoints need to be updated in this phase.

</decisions>

<specifics>
## Specific Ideas

- The observed bug is that, after installation, executing skills still creates planning docs in English.
- The expected outcome is Korean-first generated docs such as CONTEXT/PLAN/SUMMARY/VERIFICATION/UAT and related planning templates.
- Parser-sensitive labels should remain intact where required for compatibility.

</specifics>
